==============================================================================
ANALYSIS — WHY FIXED-START STILL SHOWS 1 PIT
==============================================================================

The fundamental issue: the actual features_contribution at the fixed-start
second-pit window (step ~1314) is around -2.3 to -2.5, NOT -1.5 as estimated.

Evidence: If features_contribution = -1.5 (our estimate), then with direct=+2.1:
  net = +2.1 + (-1.5) = +0.6 → POSITIVE → should fire
But it doesn't fire. This means features_contribution < -2.1, i.e., ≤ -2.2.

The frozen mlp_extractor features at the specific car state at step ~1314 of
the fixed-start trajectory produce a strongly negative projection onto the pit
row weights. This is the SAME bottleneck identified in d31 — the frozen features
are not linearly separable for the pit row in this specific trajectory configuration.

Scale analysis:
  To fire with features = -2.5: need direct > 2.5 → W_pit[128] × 0.53 > 2.5
  → |W_pit[128]| > 4.72, but threshold constraint gives: W_pit[128] × 0.60 = -b
  → b/|W| = 0.60, so b = 0.60 × 4.72 = 2.83
  → Arbitrarily large W, b pairs can overcome features, BUT PPO will then
    drive the features weights to zero (overly dominant direct connection)
    and the policy may lose the ability to distinguish worn vs fresh tyres.

Actually: with W_pit[128]=-40, b=+24:
  direct at tl=0.53: -40×0.53+24 = +2.8
  net with features=-2.5: +2.8 + (-2.5) = +0.3 → barely positive
  threshold still: -40×0.60+24 = 0 ✓

So d36 COULD try W=-40, b=+24. But this is diminishing returns.
