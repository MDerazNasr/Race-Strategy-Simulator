ROOT CAUSE ANALYSIS
------------------------------------------------------------------------------

D25 PARTIALLY SOLVED D24'S PROBLEM:
  - Froze mlp_extractor.policy_net (0.000 drift) → features identical to d21
  - Used same env as d21 → no value crisis (ep_rew_mean never crashed like d23/d24)
  - Training was STABLE in Stage 0 (ep_len=1220-1310, rolling_rate=78-87%)
  - No catastrophic 23-step crash like d24

D25 STILL FAILED BECAUSE:
  The frozen features are just INPUTS to action_net. Even with frozen inputs,
  action_net can still learn different output mappings. Specifically:

  action_net.weight = shape (3, 128):
    Row 0 (throttle): TRAINABLE — learned to produce throttle=0.131 (low throttle)
    Row 1 (steer):    TRAINABLE — learned to produce steer=1.0 with std=0.124
    Row 2 (pit):      TRAINABLE — learned slightly different pit behavior

  The gradient signal for pit timing (from forced pit experiences at threshold 0.25/0.15/0.08)
  propagated into throttle and steer rows through shared optimizer updates. Over 2M steps:
    1. Throttle converged to 0.131 (very low throttle — "cautious" driving)
    2. Steer converged to 1.0 with std=0.124 (hard right with high confidence)
    3. These wrong actions still survive stochastic training (high throttle std=0.94
       covers the variance), but fail deterministically

  Stage 2 collapse (ep_len=163) happened when steer confidence increased (std=0.12)
  and the deterministic steer=1.0 started causing crashes at different track positions.

THE FUNDAMENTAL ISSUE:
  Freezing mlp_extractor.policy_net is NECESSARY but NOT SUFFICIENT.
  Features must be frozen AND the throttle/steer OUTPUT ROWS must be frozen.

  Without freezing throttle/steer rows, the gradient from:
    - Value function updates (what throttle/steer gives more reward?)
    - Pit experience gradient (pit at threshold → value update → action update)
  ...can still corrupt the throttle/steer mapping over 2M steps.

WHY D25 DIDN'T DEGRADE AS FAST AS D24:
  D24: pit_timing_reward env → value recalibration crisis immediately
       → driving destroyed in thousands of steps (shown as ep_len=23 at eval)
  D25: same env as d21 → no crisis
       → driving degraded SLOWLY over 2M steps (ep_len=1300 → 163)
       → Stage 0 was good, Stage 1/2 eventually degraded

  The frozen features bought ~1.5M steps of good behavior before action_net drifted.

------------------------------------------------------------------------------
WHAT D26 MUST DO DIFFERENTLY
------------------------------------------------------------------------------

THREE-LAYER FREEZE (the only true solution):

Layer 1 — Freeze mlp_extractor.policy_net (features):
  Same as d25. Driving features completely preserved. (0.000 drift achieved)

Layer 2 — Freeze action_net.weight[0,:] and action_net.weight[1,:] (throttle/steer):
  Freeze the throttle and steer output rows via gradient hooks.
  This guarantees the throttle/steer mapping is identical to d21's.
  The steer=1.0 collapse CANNOT happen if the steer row is frozen.

Layer 3 — Allow action_net.weight[2,:] to train (pit row):
  The ONLY trainable policy parameter. The pit row can refine pit timing.
  D21 pits at tyre_life≈0.35; the gradient from safety-net experiences
  can reinforce or refine this without affecting throttle/steer.

Additionally:
  - Also freeze log_std[0] and log_std[1] (throttle/steer stds)
  - Allow log_std[2] to train (pit exploration std)
  - Also freeze action_net.bias[0] and bias[1]
  - Allow action_net.bias[2] to train

  Value layers remain UNFROZEN: mlp_extractor.value_net and value_net
  can adapt without affecting the policy.

Implementation: gradient hooks on action_net.weight (zero rows 0 and 1),
  action_net.bias (zero elements 0 and 1), and log_std (zero elements 0 and 1).
  PLUS requires_grad=False on mlp_extractor.policy_net (cleaner approach).

Env: pit_timing_reward=True (make_env_pit_d23) OR standard (make_env_pit)?
  With frozen throttle/steer + frozen features, the value crisis from
  pit_timing_reward CANNOT destroy driving — all driving parameters are locked.
  Consider using pit_timing_reward=True to add timing gradient to pit row.
  The value function will recalibrate (crisis period) but policy won't regress.

Expected:
  - Driving: GUARANTEED identical to d21 (all driving params frozen)
  - Pitting: should maintain d21's 1 pit/episode behavior
  - Timing: might improve slightly with pit_timing_reward gradient
  - Reward: ≈ d21's 1877 (driving preserved) or better if pit timing improves

------------------------------------------------------------------------------
LESSONS LEARNED
------------------------------------------------------------------------------
1. Freezing features (mlp_extractor.policy_net) is NECESSARY but NOT SUFFICIENT.
   Output rows (throttle/steer) must ALSO be frozen to prevent drift over 2M steps.

2. Action_net row gradient bleeding: when training pit row, the shared Adam
   optimizer state for throttle/steer rows still sees gradient signal (from value
   function updates), which can slowly corrupt those rows over 2M steps.

3. Stochastic training masks deterministic failures. D25 had ep_len=1300 in Stage 0
   (stochastic) but ep_len=163 in Stage 2 (when steer std collapsed to 0.12 and
   the bad deterministic steer became visible).

4. The freeze must cover ALL policy dimensions EXCEPT the one we want to change.
   For pit timing: freeze everything EXCEPT action_net.weight[2,:], bias[2], log_std[2].

5. The frozen mlp_extractor.policy_net approach (requires_grad=False) works PERFECTLY
   — achieved 0.000 drift. Combine this with gradient hooks on action_net rows 0,1.
