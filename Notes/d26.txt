━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

D26 is the culmination of the Week 5 freeze-to-preserve-pitting strategy. After
three failed attempts (d22–d25), we identified that fine-tuning from d21 consistently
either (a) loses pitting [d22, d23] or (b) destroys driving [d24, d25].

The root cause was always gradient bleeding:
  - d22: all params trainable → forgot pitting
  - d23: only pit row frozen → features/output drifted
  - d24: loaded into pit_timing_reward env → value crisis → steer hard-left
  - d25: features frozen but action_net output rows still trainable → steer hard-right

D26 response: THREE-LAYER FREEZE. Lock every driving parameter:

  Layer 1 — mlp_extractor.policy_net (18,176 params):
    requires_grad=False on all weights/biases of the feature extractor.
    The hidden representations encoding track state are LOCKED at d21's values.

  Layer 2 — action_net rows 0,1 (throttle/steer output, gradient hooks):
    register_hook on action_net.weight → zero gradient for rows 0 and 1.
    register_hook on action_net.bias → zero gradient for elements 0 and 1.
    Adam optimizer sees zero gradient → no update to throttle/steer output mapping.

  Layer 3 — log_std dims 0,1 (exploration std, gradient hook):
    register_hook on log_std → zero gradient for dims 0 and 1.
    Exploration std for throttle/steer LOCKED at d21's values.

ONLY TRAINABLE POLICY PARAMETERS (130 total):
  action_net.weight[2,:]   128 params — pit signal weight row
  action_net.bias[2]         1 param  — pit signal bias
  log_std[2]                 1 param  — pit signal exploration std

VALUE FUNCTION: fully trainable (mlp_extractor.value_net + value_net).

ENVIRONMENT: make_env_pit (standard, NO pit_timing_reward) — same as d21.
CURRICULUM: STAGES_PIT_V5 (safety net: threshold 0.25 → 0.15 → 0.08)
STARTING POINT: ppo_pit_v4.zip (d21, reward=1877, 1 pit)
LR: 1e-4 → 1e-6 cosine over 2M steps

ROOT CAUSE ANALYSIS: ADAM OPTIMIZER BLEEDING (d25 lesson confirmed)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  Why 6.1e-4 drift when hooks zero the gradient? Two effects:

  1. Numerical precision: Even with gradient=0, fp16/bf16 arithmetic introduces tiny
     noise. But we're on CPU fp32, so this is ~1e-7 per step.

  2. ACTUAL cause: Adam optimizer momentum states. When a parameter has requires_grad=True
     (which hook targets do), Adam still maintains m (first moment) and v (second moment)
     accumulators. If the gradient is exactly 0, the update is:
       Δw = lr * m / (sqrt(v) + eps)
     where m decays slowly (m = β1*m_prev + 0 → approaches 0 but never reaches it in
     finite steps). Over 2M steps, the residual m from before hooks were applied can
     cause drift of O(1e-4).

  Fix for future: use requires_grad=False for throttle/steer rows instead of hooks,
  IF SB3 supports per-parameter requires_grad in action_net. Gradient hooks are
  sufficient here (6.1e-4 drift is tiny and didn't affect behavior), but future d27+
  work could switch to requires_grad=False for cleaner freeze.
