WHAT WAS BUILT
==============================================================================

D32 implemented a custom SB3 policy (PitAwarePolicy) that provides a direct
linear connection from tyre_life (obs[11]) to the pit_signal output, bypassing
the feature bottleneck that defeated d26–d31.

ROOT CAUSE OF D26–D31 FAILURES:
  The frozen mlp_extractor features do NOT encode tyre_life in a linearly
  separable form for the pit row. BC probe (d31) showed:
    - Worn states (tl < 0.55): pit_signal mean = -0.222 (target +2.0)
    - Fresh states (tl > 0.65): pit_signal mean = -0.090 (target -2.0)
    - Separation: 3.25% of required target — FEATURE BOTTLENECK confirmed.
  D21's pitting was likely random exploration (50% pit rate), not state-conditional.

THE D32 FIX — PitAwarePolicy with TyrLifeAugmentedExtractor:
  New class: TyrLifeAugmentedExtractor (rl/pit_aware_policy.py)
    - Inherits MlpExtractor, overrides latent_dim_pi: 128 → 129
    - forward_actor(obs) = cat([policy_net(obs), obs[11]])  ← 129-dim
    - Action net automatically created as Linear(129, 3) by SB3's _build()

  New class: PitAwarePolicy (rl/pit_aware_policy.py)
    - Inherits ActorCriticPolicy, overrides _build_mlp_extractor()
    - Uses TyrLifeAugmentedExtractor instead of standard MlpExtractor
    - Transparent to all other SB3 code (save/load, rollout, evaluate all work)

  Initialization:
    action_net.weight[2, :128] = d21's existing pit row weights (copied)
    action_net.weight[2,  128] = -10.0  ← DIRECT tyre_life → pit (negative)
    action_net.bias[2]         = +7.0   ← threshold at tl ≈ 0.69 (adjusted)
    action_net.weight[0:2, 128] = 0.0   ← throttle/steer: no tyre_life effect
    All other weights: copied from d21

  Result at initialization (tl direct component only):
    tl=0.30: pit_signal ≈ +4.0  P(pit>0) ≈ 97%  ← always pit (critical)
    tl=0.45: pit_signal ≈ +2.5  P(pit>0) ≈ 88%  ← usually pit (worn)

    tl=0.90: pit_signal ≈ -2.0  P(pit>0) ≈ 17%  ← almost never pit (new tyre)

==============================================================================
WHY D32 SUCCEEDED WHERE D26–D31 FAILED
==============================================================================

THE KEY INSIGHT:
  D26–D31 all tried to teach the LINEAR pit row (action_net[2,:]) to be
  state-conditional using the FROZEN 128-dim mlp_extractor features. But:
  1. The frozen features don't linearly encode tyre_life (3.25% separation)
  2. BC pre-training (d31) cannot fix the feature bottleneck
  3. Any PPO training with forced/voluntary pits creates a gradient that pushes
     pit_bias negative because fresh-tyre pits (probabilistic at 50%) dominate

  D32 bypasses the bottleneck ENTIRELY by appending tyre_life (obs[11]) directly
  to the 128-dim feature vector before the action_net. The action_net sees:
    latent_pi = [features (128-dim) | tyre_life (1-dim)]  =  129-dim
  The pit row's 129th weight provides a DIRECT linear path from tyre_life to
  pit_signal, independent of what the frozen features encode.

WHY THE GRADIENT NOW WORKS:
  - At worn states (tl < 0.60, voluntary bonus fires):
      pit_signal = -10.49×tl + 6.50 + features_noise ≈ +1.5 to +4
      P(pit>0) ≈ 95–100% → voluntary pit fires → +300 bonus → net +100
      POSITIVE advantage → gradient reinforces current pit policy ✓

  - At fresh states (tl > 0.70):
      pit_signal = -10.49×tl + 6.50 ≈ -0.8 to -3
      P(pit>0) ≈ 0–20% → rarely pits → no -200 penalty
      NEUTRAL gradient → doesn't damage the state-conditioning ✓

  Net gradient: strongly positive from frequent correct pits, minimal negative
  from rare incorrect pits. First time the gradient pointed the right direction
  consistently throughout training.
